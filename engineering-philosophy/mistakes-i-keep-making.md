# Mistakes I Keep Making

Self-awareness is the foundation of growth. These are the engineering traps I frequently fall into, along with the rules I have created to protect myself from my own habits.

---

## 1. Building Before Designing

### The Mistake
Because I love coding, I often get excited about a feature and immediately open VS Code to start implementation. 

### The Consequence
Without a clear design, I inevitably run into problems halfway through. I end up:
*   Rewriting Express / ASP.NET API routes because the request flows are clunky.
*   Renaming database models three times in the same day, resulting in a pile of local migration headaches.
*   Writing spaghetti adapters to stitch together domains that should have been decoupled from the start.

### My Fix
I now force myself to spend **at least 30 minutes designing on paper or a markdown file before writing a single line of code**. I outline:
1.  The HTTP request/response JSON payloads.
2.  The database tables and relations.
3.  The main service interfaces and their exact dependencies.

---

## 2. Optimizing Too Early

### The Mistake
I catch myself thinking about scale before I even have a single user. I worry about how a system will handle millions of concurrent queries, or write complex caching layers for data that rarely changes.

### The Consequence
I spend days configuring Redis cache-aside models, writing database sharding configurations, or splitting code into multi-layer architectures. This over-engineering introduces unnecessary maintenance overhead, bugs, and configuration drift, delaying the initial launch.

### My Lesson
**A simple solution built today beats a highly scalable architecture that nobody uses.** I now write the simplest code that solves the immediate business requirement. I build with clean domain boundaries so that *when* scaling problems arise, the module can be easily refactored without breaking the rest of the application.

---

## 3. Trusting AI Code Too Quickly

### The Mistake
Because modern LLMs generate syntactically correct code that compiles cleanly, I used to copy-paste generated utility methods or SQL queries assuming they were correct.

### The Consequence
I encountered silent errors in production:
*   A query generated by an AI for **SmartFood** retrieved entire database collections into web server memory to perform filters, causing system slowdowns.
*   Auth validations generated by AI for JWT checks sometimes bypassed security roles under certain malformed payloads (elevation of privilege).
*   AI code often skips writing logs or telemetry, making the feature difficult to debug when it breaks.

### My Fix
I treat all AI output as if it were written by a junior intern who is trying to cut corners. Before accepting any code:
1.  I manually read and explain every line of the generated snippet.
2.  I write unit tests targeting the failure states (e.g., passing `null`, invalid strings, or extreme values).
3.  I run the query prefixed with `EXPLAIN` in PostgreSQL to inspect its database execution plan.
